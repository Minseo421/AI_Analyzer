# Repository Analysis Framework

This document defines manual repository-level governance coding for the AI-use disclosure research study.

Repository governance analysis is performed by researchers. It is not generated by the PR analysis script. The script may provide PR-level evidence and preliminary compliance summaries, but policy coding must be completed manually from repository artifacts.

## General Repository Coding Principles

- Code observable governance evidence, not assumed project intent.
- Prefer stable repository URLs over screenshots or search results.
- Record the source artifact and date of access when possible.
- Distinguish policy visibility from policy enforcement.
- Treat absent evidence as `None found` or `0`, not as proof that no informal governance exists.
- Use notes for borderline cases and organization-level policies.

## Policy Location

Definition:

The repository artifact where AI-use disclosure guidance appears.

Allowed values:

- Free text, such as `CONTRIBUTING.md`, `PULL_REQUEST_TEMPLATE.md`, `README.md`, `docs/`, `SECURITY.md`, `Maintainer comment`, `Organization policy`, `None found`.

Coding rules:

- Code the most contributor-relevant location.
- Prefer PR-workflow locations over general documentation when both exist.
- Use `None found` when no AI-use disclosure policy is found after review.

Examples:

- `PULL_REQUEST_TEMPLATE.md` when the PR template asks whether AI was used.
- `CONTRIBUTING.md` when the contributing guide requires AI disclosure.
- `Maintainer comment` when policy appears only in recurring PR review comments.

Manual analysis instructions:

- Inspect contributor guides, PR templates, issue templates, docs, CI workflows, and recent maintainer guidance.
- Record `Policy URL` separately in the policy tracker when available.

## Disclosure Mechanism

Definition:

The workflow mechanism used to request, prompt, or enforce AI-use disclosure.

Allowed values:

- `Documentation`
- `PR template`
- `Checklist`
- `CI check`
- `Maintainer review`
- `Bot/app`
- `Organization policy`
- `None found`
- Free text for mixed or unusual mechanisms.

Coding rules:

- Code every visible mechanism in notes if multiple are present.
- For the main field, choose the mechanism closest to the contributor's PR submission workflow.
- Do not code a mechanism as enforcement unless it creates a prompt, check, or review action.

Examples:

- A PR checklist item asking `Was AI used?` -> `Checklist`.
- A required CI job checking disclosure text -> `CI check`.
- A paragraph in `CONTRIBUTING.md` only -> `Documentation`.

Manual analysis instructions:

- Open the actual PR creation template when possible.
- Check whether the mechanism is still active on the default branch.

## Visibility Score

Definition:

How easy it is for contributors to encounter the AI disclosure policy before or during PR submission.

Allowed values:

- `0`, `1`, `2`, `3`, `4`

Coding rules:

| Score | Rule |
| --- | --- |
| `0` | No AI disclosure policy found. |
| `1` | Policy exists but is difficult to find, informal, stale, or only in scattered discussion. |
| `2` | Policy appears in repository documentation but not in the PR workflow. |
| `3` | Policy appears in contributor-facing documentation or PR templates. |
| `4` | Policy is highly visible at the point of contribution, such as a required PR checklist item, prominent contributing guide section, or automated prompt. |

Examples:

- `0`: no AI policy found.
- `2`: AI disclosure paragraph in docs.
- `4`: PR template asks contributors to answer an AI-use question.

Manual analysis instructions:

- Start from the perspective of a new contributor opening a PR.
- Record uncertainty when policy visibility differs across branches or templates.

## Enforcement Score

Definition:

How strongly the repository operationalizes, checks, or reinforces AI disclosure expectations.

Allowed values:

- `0`, `1`, `2`, `3`, `4`

Coding rules:

| Score | Rule |
| --- | --- |
| `0` | No enforcement mechanism found. |
| `1` | Policy is advisory only, with no evidence of checking. |
| `2` | Maintainers may review disclosure manually, but no structured mechanism is visible. |
| `3` | Disclosure is requested through a structured PR template, checklist, or contribution workflow. |
| `4` | Disclosure is operationalized through mandatory checklist completion, CI/app checks, blocking review practice, or repeated maintainer enforcement. |

Examples:

- `1`: contributing guide asks for disclosure but no PR prompt exists.
- `3`: PR template includes an AI disclosure checkbox.
- `4`: CI blocks missing disclosure or maintainers consistently require correction before merge.

Manual analysis instructions:

- Look for evidence in PR templates, CI checks, bot comments, and maintainer review behavior.
- Do not infer enforcement from policy text alone.

## Transparency

Definition:

Whether policy language emphasizes transparency about AI assistance.

Allowed values:

- `1` present
- `0` absent
- blank if uncoded

Coding rules:

- Code `1` when the policy says contributors should disclose, identify, reveal, or be transparent about AI use.
- Code `0` when transparency is not a stated rationale.

Examples:

- `Please disclose any AI-generated or AI-assisted contributions.` -> `1`.
- `AI code must pass tests.` with no disclosure rationale -> usually `0`.

Manual analysis instructions:

- Code the rationale present in policy wording, not the reviewer's interpretation.

## Accountability

Definition:

Whether policy language emphasizes contributor responsibility for AI-assisted work.

Allowed values:

- `1` present
- `0` absent
- blank if uncoded

Coding rules:

- Code `1` when contributors are told they remain responsible for AI-generated or AI-assisted code.
- Code `0` when responsibility/accountability is not mentioned.

Examples:

- `You are responsible for all AI-generated code you submit.` -> `1`.
- `AI use must be disclosed.` without responsibility language -> `0` for this field.

Manual analysis instructions:

- Separate accountability from quality and legal rationales unless the policy states both.

## Reviewer Burden

Definition:

Whether policy language mentions reducing reviewer workload or improving reviewability.

Allowed values:

- `1` present
- `0` absent
- blank if uncoded

Coding rules:

- Code `1` when policy links disclosure to helping maintainers review, triage, or assess generated changes.
- Code `0` when reviewer burden is not mentioned.

Examples:

- `Disclose AI assistance so reviewers can focus extra attention on generated areas.` -> `1`.

Manual analysis instructions:

- Do not code reviewer burden merely because review is implied by PR workflow.

## Code Quality

Definition:

Whether policy language links AI use to correctness, maintainability, testing, or quality control.

Allowed values:

- `1` present
- `0` absent
- blank if uncoded

Coding rules:

- Code `1` when policy discusses tests, correctness, security, maintainability, hallucinations, or quality risks of AI-assisted code.
- Code `0` when quality is not a stated rationale.

Examples:

- `AI-generated code must be carefully tested and reviewed for correctness.` -> `1`.

Manual analysis instructions:

- Code based on explicit language, not general assumptions about AI risk.

## Trust / Provenance

Definition:

Whether policy language emphasizes trust, provenance, authorship, traceability, or origin of contributions.

Allowed values:

- `1` present
- `0` absent
- blank if uncoded

Coding rules:

- Code `1` when policy asks contributors to identify AI-generated portions, authorship, provenance, or origin.
- Code `0` when trust/provenance is absent.

Examples:

- `Mark which parts were generated by AI so maintainers can trace authorship.` -> `1`.

Manual analysis instructions:

- Distinguish provenance from simple transparency when possible; code both if both rationales are present.

## Legal / Licensing

Definition:

Whether policy language mentions licensing, copyright, IP, legal risk, or provenance obligations.

Allowed values:

- `1` present
- `0` absent
- blank if uncoded

Coding rules:

- Code `1` when policy references copyright, licensing compatibility, IP ownership, training data concerns, or legal liability.
- Code `0` when legal/licensing concerns are absent.

Examples:

- `Do not submit AI-generated code unless you can confirm licensing and copyright compatibility.` -> `1`.

Manual analysis instructions:

- Keep legal/licensing separate from general quality concerns.
- Record exact policy language in notes when the legal rationale is ambiguous.
